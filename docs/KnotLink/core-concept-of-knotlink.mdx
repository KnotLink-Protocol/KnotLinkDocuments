# KnotLink 核心概念

## 1. KnotLinkClient_PyQt 友好型简介

| TIP：KnotLinkClient依赖服务端，安装服务端后才能正常使用。 |
| --------------------------------------------------------- |

KnotLinkClient_PyQt 是一个基于 PyQt5 的客户端库，用于实现信号通信和 Socket（API） 交互。它支持两种主要的通信模式：**发布-订阅模式** 和 **请求-响应模式**。这两种模式分别适用于不同的应用场景，通过理解这两种模式，你可以更好地选择和使用 KnotLinkClient_PyQt 的功能。

## 2. 通信模式概述

### 2.1 发布-订阅模式（Publish-Subscribe Pattern）

想象一下，你有一个朋友（我们叫他 **A**），他喜欢在社交媒体上分享他的日常动态。他的朋友们（我们叫他们 **订阅者**）可以选择关注他的动态。每当 **A** 发布一条新动态时，所有关注他的朋友都会立即收到通知。

- **核心特点**：
  - **一对多通信**：就像 **A** 有一个动态，所有关注他的朋友都能看到。
  - **解耦**：**A** 不需要知道谁关注了他，他只是发布动态；朋友们也不需要告诉 **A** 他们关注了他，他们只是接收动态。
  - **实时性**：一旦 **A** 发布了动态，所有关注他的朋友都能立即看到。
  - **灵活性**：朋友们可以随时关注或取消关注 **A**，而 **A** 不需要做任何改变。
- **应用场景**：
  - **实时数据更新**：就像股票价格更新，所有关注股票的人都能立即看到价格变化。
  - **事件通知**：比如系统状态变化，所有关注系统的用户都能收到通知。
  - **多客户端数据共享**：比如多人在线游戏，所有玩家都能实时收到游戏状态更新。

### 2.2 请求-响应模式（Request-Response Pattern）

再想象一下，你有一个朋友（我们叫他 **B**），他是个万事通，你可以向他提问，他会回答你的问题。你和你的朋友们（我们叫他们 **询问者**）都可以向 **B** 提问，**B** 会逐一回答。

- **核心特点**：
  - **多对一通信**：就像你和你的朋友们都可以向 **B** 提问，但 **B** 只有一个人回答。
  - **同步通信**：你问了问题后，会等待 **B** 的回答，不会去做别的事情，直到收到回答。
  - **交互性**：你和 **B** 之间有明确的问答逻辑。
  - **可靠性**：**B** 会确保每个问题都能得到回答，如果他没回答，你可能会再问一次。
- **应用场景**：
  - **查询数据库**：就像你向数据库提问，数据库会返回结果。
  - **获取配置信息**：比如你向服务器请求配置信息，服务器会返回这些信息。
  - **客户端与服务器交互**：比如你向服务器请求数据，服务器会返回你需要的数据。

## 3. 四大模块介绍

### 3.1 SignalSender（信号发送者）

想象一下，**A** 是一个广播员，他有一个麦克风，可以向所有人广播消息。

- **功能**：负责生成和发送信号。
- **特点**：
  - **角色固定**：就像 **A** 是那个广播员，他的角色是固定的。
  - **异步广播**：**A** 发送消息后，不需要等待任何人回应，他可以继续做别的事情。
  - **解耦**：**A** 不需要知道谁在听他的广播，他只是发送消息。

### 3.2 SignalSubscriber（信号订阅者）

想象一下，你是 **A** 的一个朋友，你关注了 **A** 的动态，每当 **A** 发布新动态时，你都会收到通知。

- **功能**：负责接收来自 SignalSender 的信号。
- **特点**：
  - **角色灵活**：就像你可以随时关注或取消关注 **A**，你的角色是灵活的。
  - **实时接收**：一旦 **A** 发布了新动态，你就能立即看到。
  - **动态订阅/取消订阅**：你可以随时决定关注或取消关注 **A**，而 **A** 不需要知道。

### 3.3 OpenSocketQuerier（请求者）

想象一下，你是向 **B** 提问的人，你有问题，你会向 **B** 提问，然后等待他的回答。

- **功能**：负责向响应者发送查询请求。
- **特点**：
  - **角色灵活**：就像你可以随时向 **B** 提问，你的角色是灵活的。
  - **同步等待响应**：你问了问题后，会等待 **B** 的回答，不会去做别的事情。
  - **交互性强**：你和 **B** 之间有明确的问答逻辑。

### 3.4 OpenSocketResponser（响应者）

想象一下，**B** 是那个万事通，他会回答所有向他提问的人。

- **功能**：负责处理请求并返回响应。
- **特点**：
  - **角色固定**：就像 **B** 是那个万事通，他的角色是固定的。
  - **处理多个请求**：**B** 可以同时回答多个问题。
  - **高可靠性**：**B** 会确保每个问题都能得到回答，如果他没回答，提问的人可能会再问一次。

## 4. 选择适合你的模式和模块

- **如果你需要实时广播数据或事件通知**，并且希望发送者和接收者之间解耦，那么 **发布-订阅模式** 是你的最佳选择。你可以使用 **SignalSender** 作为信号发送者，使用 **SignalSubscriber** 作为信号订阅者。
- **如果你需要客户端与服务器之间的交互**，并且需要确保每个请求都能得到响应，那么 **请求-响应模式** 更适合你。你可以使用 **OpenSocketQuerier** 作为请求者，使用 **OpenSocketResponser** 作为响应者。

- 微博热搜：大 V（Sender）发博，所有粉丝（Subscriber）瞬间刷到。

⚠️ 注意事项（新手常踩坑）

- 听众可以同时存在很多个
  你可以开 100 个订阅脚本，都能同时收到同一条广播。
- 广播一次，全员可见
  一嗓子下去，不会漏掉任何一个人。

### 请求-响应模式（OpenSocketQuerier + OpenSocketResponser）

新手一句话教程

------

#### 🎯 一句话总结

**“一个人负责回答，其余人负责提问。”**

- 回答者：OpenSocketResponser（固定身份，一直在等别人来问）。
- 提问者：OpenSocketQuerier（谁想提问就谁来，问完就走）。

#### 📌 场景举例（秒懂）

- **客服系统**：一个客服小姐姐（Responser）坐班，所有用户（Querier）随时提问。
- **图书馆咨询台**：咨询员永远在柜台，读者轮流去问“有没有《三体》？”
- **智能音箱后台**：音箱硬件（Querier）不停地问云端（Responser）“今天天气？”

------

#### ⚠️ 注意事项（新手常踩坑）

1. **先开客服，再去提问**
   否则提问者会连不上，就像客服还没上班你就打电话。
2. **APPID 和 OpenSocketID 必须一致**
   相当于“房间号”，客服和提问者要进同一个房间才能对话。
3. **提问者可以同时存在很多个**
   你可以开 10 个提问脚本，客服小姐姐会按顺序一个个回答。
4. **回答一次只能回一个人**
   客服一次只能回一条消息，别指望一条答案飘给十个人。

## 数据格式标准化

无论是发送信号还是发起请求，我们建议您采用标准化的数据格式。这不仅有助于他人快速理解数据内容，还能有效降低项目维护的复杂度。

以下是两种推荐格式，它们分别适用于不同的需求场景：

### 键值对

:::info[键值对（Key-Value Pairs）]

键值对：最简单的数据交换格式之一，仅由键和值组成，通常以简单的分隔符（如冒号或等号）分隔。示例： 

```key1=value1;key2=value2```
:::


JSON

:::info[JSON（JavaScript Object Notation）]

JSON：一种轻量级的数据交换格式，易于人阅读和编写，同时也易于机器解析和生成。它支持复杂的数据结构，如嵌套的对象和数组。JSON由键值对组成，键和值之间用冒号分隔，不同的键值对之间用逗号分隔，并且整个结构由大括号包围。示例：

```json
{
  "key1": "value1",
  "key2": "value2",
  "key3": ["item1", "item2", "item3"],
  "key4": {
    "subKey1": "subValue1",
    "subKey2": "subValue2"
  }
}
```

:::
